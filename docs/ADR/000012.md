# トレーニング可否チェックのバグ修正と再トレーニング実行結果

## ADR's STATUS

採用済み (2026-03-02)

## CONTEXT

ADR-0011 で導入した ML モデル再トレーニングスケジューラを初めて weekly モードで手動実行したところ、`/retrain/check` エンドポイントが 500 Internal Server Error を返した。調査の結果、`ml/app/training/checks.py` に 2 件のバグが存在し、トレーニング可否チェックが正しく動作しない状態だった。

### 発見されたバグ

1. **存在しないカラム参照**: HRV トレーニング可否チェックで `ds.ln_rmssd_mean` を参照しているが、`daily_summaries` テーブルにこのカラムは存在しない。正しいカラム名は `ds.hrv_daily_rmssd`
2. **completeness_pct の閾値不整合**: `daily_data_quality.completeness_pct` は 0.0〜1.0 のスケール（1.0 = 100%）で格納されているが、チェックコードでは `avg_completeness < 40` と比較しており、常に条件を満たさない（最大値 1.0 < 40 は常に true）。3 モデルすべてのチェックに影響

バグ 2 により、直近 7 日間のデータ品質がどれだけ良好でもすべてのモデルが `recent_quality_ok: false` となり、Anomaly と Divergence のトレーニングがブロックされていた。

## DECISION MAKING

### 修正内容

| バグ | 修正前 | 修正後 | 影響箇所 |
|---|---|---|---|
| カラム名不正 | `ds.ln_rmssd_mean IS NOT NULL` | `ds.hrv_daily_rmssd IS NOT NULL` | `checks.py` L111, L131 |
| 閾値スケール不整合 | `avg_completeness < 40` | `avg_completeness < 0.40` | `checks.py` L75, L155, L232（全 3 モデル） |

フォーマット文字列も `:.0f` → `:.0%` に変更し、0.95 が `95%` と表示されるようにした。

### 修正方針

- `completeness_pct` の格納値を 0-100 スケールに変更する案も検討したが、既存の全クエリが 0-1 スケールを前提としているため、チェックコード側の閾値を合わせる方が影響範囲が最小
- `ln_rmssd_mean` は開発中の命名揺れが残ったもの。実際のトレーニングコード（`hrv_features.py` 等）では正しく `hrv_daily_rmssd` を使用しており、チェックコードのみが不整合だった

## RESULTS, EFFECTS

### 再トレーニング結果（weekly モード、2026-03-02 実行）

| モデル | ステータス | データ量 |
|---|---|---|
| Anomaly (Isolation Forest) | success | 172 valid days |
| HRV (XGBoost + LSTM) | skipped | 21 valid days（閾値 90 日未達） |
| Divergence (Ridge + CuSum) | success | 76 paired observations |

所要時間: 1.3 秒（HRV スキップのため高速完了）

### モデル精度

#### Anomaly Detection

| 指標 | 今回 (v1772413965) | 前回 (v1771942083) |
|---|---|---|
| 学習日数 | 172 | 167 |
| contamination | 0.02 | 0.02 |
| POT threshold | -0.023 | -0.042 |

POT 閾値が上昇（-0.042 → -0.023）。5 日分の新データにより正常範囲の境界がやや緩和。教師なし学習のため R2/MAE は非該当。

#### Divergence Detection

| 指標 | 今回 (v1772413965) | 前回 (v1771942290) | 変化 |
|---|---|---|---|
| 学習ペア数 | 76 | 71 | +5 |
| R2 | 0.243 | 0.169 | +44% |
| MAE | 16.2 | 17.7 | -8% |
| RMSE | 20.4 | 21.2 | -4% |

全指標で改善。R2 = 0.24 は絶対値としては低いが、主観スコアと客観バイオメトリクスの乖離検出という性質上、ターゲット変数のノイズが大きいタスクである。

#### HRV Prediction（スキップ — 前回モデル継続使用）

| 指標 | 継続中モデル (v1771942248) |
|---|---|
| CV MAE | 0.596 |
| CV RMSE | 0.978 |
| CV R2 | 0.348 |
| Directional Accuracy | 79.0% |

`hrv_daily_rmssd IS NOT NULL` を満たす日が 21 日のみ（閾値 90 日）。Fitbit の HRV データは SpO2 センサー依存で欠損が多いことが原因。

### PROS

- バグ修正により、トレーニング可否チェックが設計どおりに動作するようになった
- 直近データ品質が良好な場合に Anomaly / Divergence が正しくトレーニング対象と判定される
- 再トレーニング後の予測エンドポイント（VRI, Anomaly）の正常動作を確認済み

### CONS, TRADEOFF

- HRV モデルは `hrv_daily_rmssd` の欠損が多くトレーニングできない状態が続く。HRV データの蓄積を待つか、閾値の引き下げを検討する余地がある
- Divergence の R2 = 0.24 は改善傾向にあるが、引き続きデータの蓄積による精度向上が必要

### 検証手順

```bash
# 1. トレーニング可否チェック
curl -s http://localhost:8880/api/retrain/check | jq .

# 2. weekly モードで再トレーニング
curl -s -X POST http://localhost:8880/api/retrain/trigger \
  -H 'Content-Type: application/json' \
  -d '{"mode": "weekly"}' | jq .

# 3. 結果確認
curl -s http://localhost:8880/api/retrain/status | jq .
curl -s http://localhost:8880/api/retrain/logs | jq .

# 4. 予測エンドポイント動作確認
curl -s http://localhost:8880/api/vri?date=2026-03-01 | jq .
curl -s http://localhost:8880/api/anomaly?date=2026-03-01 | jq .
```
