# LLM アドバイスエンドポイントのタイムアウト短縮とフォールバック応答の導入

## ADR's STATUS

採用済み (2026-04-09)

## CONTEXT

トップページのロードが断続的にタイムアウトする障害が発生した。調査の結果、以下の因果関係が判明した。

1. Ollama コンテナが長時間稼働後に GPU アクセスを喪失し、全レイヤーが CPU にフォールバックした
2. CPU のみでの LLM 推論は 2 分以内に完了できず、Ollama が毎回タイムアウト後に 500 を返却
3. ML サービス → Go API → フロントエンド SSR の各層で 120 秒のタイムアウトが連鎖し、`/api/advice` が約 2 分間ブロック
4. Go API が advice 生成失敗時に HTTP 500 を返すため、フロントエンドがリトライループに入った

GPU 喪失は Docker の cgroup ドライバ (systemd) と NVIDIA Container Toolkit の既知の非互換性が原因。systemd のリロード時にコンテナ内の GPU デバイスノードが無効化される (NVIDIA/nvidia-container-toolkit#48)。

advice 以外のエンドポイント (biometrics, conditions, sleep, heartrate 等) は全て 1-10ms で正常に応答しており、影響は advice エンドポイントに限定されていた。

## DECISION MAKING

インフラ対策とソフトウェア対策の両面で対応する。

### インフラ対策

- Ollama コンテナの再起動で GPU を即時復旧
- Docker の cgroup ドライバを `cgroupfs` に変更し、systemd リロード時の GPU デバイス喪失を防止

### ソフトウェア対策 (防御的改善)

GPU 問題が再発した際にページ全体への影響を最小化するため、タイムアウトチェーンを短縮しエラーハンドリングを改善する。

| 層 | 変更前 | 変更後 |
|---|---|---|
| ML (Ollama 呼び出し) | 120 秒 | 30 秒 |
| Go API (ML 呼び出し) | 120 秒 | 35 秒 |
| フロントエンド SSR | 120 秒 | 40 秒 |

各層のタイムアウトに 5 秒のマージンを設け、内側の層から順にタイムアウトする設計とした。

Go API の advice ハンドラで ML クライアントエラー時に HTTP 500 ではなくフォールバック応答 (HTTP 200 + プレースホルダーテキスト) を返すようにした。これにより、LLM が利用不可能な場合でもフロントエンドがエラーループに入ることを防止する。

ML サービスのエラーログで `%s` を `%r` に変更し、httpx.HTTPError の空文字列表現ではなく repr 表現でエラー詳細を出力するようにした。

検討した代替案:
- タイムアウトを全層で同一 (30 秒) にする → 外側がタイムアウトすると内側のリソースがリークするため、段階的マージンを採用
- advice エンドポイントを完全に非同期化 (キュー + ポーリング) → 現時点では SSR の Promise ストリーミングで十分機能しているため過剰

## RESULTS, EFFECTS

### 変更ファイル

| ファイル | 変更 |
|---|---|
| `ml/app/config.py` | `ollama_timeout` デフォルト値 120.0 → 30.0 |
| `ml/app/routers/advice.py` | エラーログの書式を `%s` → `%r` に変更 |
| `api/adapter/mlclient/client.go` | GetAdvice / RegenerateAdvice の HTTP タイムアウトを 120s → 35s |
| `api/handler/advice.go` | ML エラー時に 500 → 200 フォールバック応答を返却 |
| `frontend/src/lib/server/services/dashboard.ts` | advice フェッチのタイムアウトを 120s → 40s |

### PROS

- GPU 喪失時の影響がページ全体のタイムアウト (2 分) から advice カードのみのフォールバック表示 (30 秒以内) に局所化
- エラーログに httpx.HTTPError の詳細が記録され、障害原因の特定が容易に
- フロントエンド SSR の advice Promise ストリーミング設計は変更不要で、既存の `{#await}` による段階的表示がそのまま機能

### CONS, TRADEOFF

- GPU 正常稼働時でも LLM 推論が 30 秒を超えるとタイムアウトする。現行モデル (gemma4-e4b-q4km) の GPU 推論は通常 15-27 秒で完了するため実用上問題ないが、より大きなモデルへの移行時にはタイムアウト値の再調整が必要
- advice エラー時にフォールバック応答 (HTTP 200) を返すため、監視システムがエラー率でアラートを出す場合は別途ログベースのモニタリングが必要

## APPENDIX

### GPU 喪失時のログシーケンス

```
# 起動時 (正常)
offloaded 43/43 layers to GPU
model weights: CUDA0 = 5.0 GiB

# 約15時間後 (GPU 喪失)
ggml_cuda_init: failed to initialize CUDA: no CUDA-capable device is detected
offloaded 0/43 layers to GPU
model weights: CPU = 5.5 GiB
```

### Docker cgroup 恒久対策

`/etc/docker/daemon.json` に `"exec-opts": ["native.cgroupdriver=cgroupfs"]` を追加し Docker を再起動する。
