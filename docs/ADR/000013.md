# 1日アドバイス生成の途中切れ修正（Ollama トークン上限・後処理截断の見直し）

## ADR's STATUS

採用済み (2026-04-05)

## CONTEXT

「今日の一言」ヘルスアドバイス（`/api/advice`）が毎回途中で切れる問題が報告された。システムプロンプトは 500〜1000 文字の日本語出力を要求しているが、実際には 350〜680 文字程度で途切れ、【今日のアクション】や締めの一言が欠落する状態だった。

### 原因分析

2 箇所の制約が重なり、出力が短くなっていた。

| 制約 | 箇所 | 値 | 影響 |
|---|---|---|---|
| Ollama `num_predict` | `ml/app/routers/advice.py` L401 | 1024 トークン | **主因**。日本語は 1 文字あたり約 2〜3 トークンを消費するため、1024 トークンでは約 350〜680 文字しか生成できない |
| 後処理ハード截断 | `ml/app/routers/advice.py` L356 | 1200 文字 | 副因。プロンプト上限 1000 文字に対してバッファが 200 文字しかなく、モデルが自然に文を終える余地が不足 |

Go API 層（`api/handler/advice.go`, `api/adapter/mlclient/client.go`）およびデータベース（`daily_advice.advice_text TEXT`）には長さ制限なし。問題は ML サービス内に限定される。

### 日本語トークン効率の背景

使用モデル `gemma4-e4b-q4km`（Gemma 4B 量子化）の日本語トークナイザは、漢字・ひらがな混在テキストで 1 文字あたり平均 2.5 トークン程度を消費する。英語（1 トークン ≈ 4 文字）と比較して効率が大幅に低い。

## DECISION MAKING

### 方針

- `num_predict` を 1024 → 2560 に引き上げ、1000 文字の日本語を安定して生成可能にする（1000 文字 × 3 トークン/文字 = 3000 が最悪ケース、2560 で大多数をカバー）
- `num_predict` を `Settings` に追加し、環境変数 `OLLAMA_NUM_PREDICT` で調整可能にする（モデル変更時にトークン効率が変わるため）
- 後処理の截断上限を 1200 → 1500 文字に緩和し、モデルが自然に文を終えられる余地を確保
- タイムアウト（120 秒）は変更不要。4B 量子化モデルで 2560 トークンは十分余裕がある

### 検討した代替案

| 案 | 不採用理由 |
|---|---|
| `num_predict` を 4096 以上に設定 | 生成時間が不必要に増加するリスク。2560 で目標範囲を十分カバーできる |
| 後処理の截断を撤廃 | LLM が指示を無視して長文を出すケースに対するセーフティネットは残すべき |
| システムプロンプトで文字数を減らす | ユーザー体験の劣化。5 セクション構成のアドバイスには 500〜1000 文字が必要 |

## RESULTS, EFFECTS

### 変更内容

| ファイル | 変更 |
|---|---|
| `ml/app/config.py` L33 | `ollama_num_predict: int = 2560` を追加 |
| `ml/app/routers/advice.py` L389 | `_call_ollama` に `num_predict` パラメータを追加 |
| `ml/app/routers/advice.py` L401 | ハードコード `1024` → `num_predict` 引数を使用 |
| `ml/app/routers/advice.py` L439 | 呼び出し元で `settings.ollama_num_predict` を渡す |
| `ml/app/routers/advice.py` L356-360 | 截断上限 `1200` → `1500`、文境界探索下限 `400` → `500` |
| `ml/tests/test_advice_router.py` | fixture に `ollama_num_predict` 追加、截断ロジックのテスト 3 件追加 |

### PROS

- 500〜1000 文字の目標範囲を安定して生成可能になり、5 セクション構成のアドバイスが完結する
- `num_predict` が環境変数で設定可能になり、モデル変更時のチューニングが容易
- 後処理のセーフティネットは維持しつつ、バッファを拡大して不要な截断を防止
- 短い出力（<400 文字）へのワーニングログは既存のまま維持

### CONS, TRADEOFF

- `num_predict` 増加（1024 → 2560）により、生成時間が最大 2.5 倍に延びる可能性がある。ただしモデルは EOS トークンで自然に停止するため、実際の増加幅はこれより小さい
- 後処理上限の引き上げ（1200 → 1500 文字）により、UI 上のアドバイス表示領域が若干長くなる場合がある

## APPENDIX

### 検証手順

```bash
# 1. テスト実行
cd ml && uv run pytest tests/test_advice_router.py -v

# 2. アドバイス生成（要 Docker 環境）
curl -s http://localhost:8880/api/advice?date=2026-04-05 | jq '.advice_text | length'
# → 500〜1000 の範囲に収まることを確認

# 3. 再生成で截断されないことを確認
curl -s -X POST "http://localhost:8880/api/advice/regenerate?date=2026-04-05" | jq '.advice_text'
```

### 数値の根拠

| パラメータ | 値 | 根拠 |
|---|---|---|
| `num_predict: 2560` | 目標 1000 文字 × 2.5 トークン/文字 = 2500。余裕を持たせて 2560 |
| 截断上限: 1500 文字 | 目標上限 1000 文字の 1.5 倍。モデルが自然に完結できるバッファ |
| 文境界下限: 500 文字 | プロンプトの最小文字数（500）と一致させ、最低限の内容を保証 |
